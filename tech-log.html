SGF Auditor — Technical log

1. Introduction

SGF Auditor is a specialized web application engineered to validate, sanitize, and analyze Smart Game Format (SGF) files. The core engine is built strictly around the global reference validation standards to ensure that game records adhere strictly to standard specifications.

The platform performs two key operations:

  1. Compliance Auditing: Executing standard syntax checking, identifying obsolete properties, checking coordinate structures, and flagging non-standard properties.
  2. Sanitization & Auto-Fixing: Re-encoding, restructuring, and repairing syntax anomalies in-browser, generating standard-compliant clean SGF assets for export.

2. SGF FF[4] Standard

The Smart Game Format (SGF) is a tree-structured text format used to store board game records. The current standard is File Format 4 (FF[4]), originally defined in 1999 by Arno Hollosi.

An SGF file consists of a tree structure made up of game nodes:

  • Game Collection: A group of GameTrees. Can contain multiple independent games.
  • Game Tree: Represents a single game sequence, consisting of Nodes and Variations.
  • Node: A collection of Properties (e.g. moves, comments, setup actions).
  • Property: Composed of a Property Identifier (e.g., SZ for board size) and one or more values enclosed in brackets (e.g., [19]).

Values in SGF have specific formats. Key data types include:

Value Type Syntax Description Examples
Number / Real Integer or floating point values. [6.5] (Komi), [19] (Size)
Color Single character: B (Black) or W (White). [B], [W]
Point Alphabetic letters indicating board coordinates. [pd], [dp]
Move Board point indicating where a stone is played, or pass. [pd], [] (Pass in FF[4])
SimpleText / Text Strings of characters, text permits newlines and spaces. [AlphaGo vs Lee Sedol]

3. Go Game Focus

While SGF FF[4] supports multiple games (such as Chess, Othello, and Backgammon), SGF Auditor focuses strictly on Go (GM[1]).

By locking our audit scope to Go, we enforce a strict set of Go-specific compliance rules:

  • Coordinates System: Points must consist of lowercase alphabetic characters in the range [a-s] for 19x19 boards. Bounds check dynamically shifts if board size (SZ) varies from 1 to 52.
  • Pass Move Notation: A pass move in SGF FF[4] is represented as an empty value []. The obsolete coordinate [tt] is permitted only for board sizes 19x19 or smaller.
  • Node Separation: Move properties (B / W) and Setup properties (AB / AW / AE) cannot exist within the same node.
  • Cardinality Rules: Game info properties (e.g., KM, RU, RE, PB, PW) must reside strictly in the Root Node and occur at most once.

4. WebAssembly & SGFC Integration

SGF Auditor is powered exclusively by the official SGFC (SGF Syntax Checker & Converter) v2.3 codebase written in C. We compiled the C sources directly into a WebAssembly module (sgfc.wasm) with JavaScript glue code (sgfc.js) using the Emscripten compiler.

Technical Detail: To enable in-browser execution, we configured Emscripten's virtual filesystem (FS). When a user drops an SGF file, JavaScript writes it to a virtual file buffer, calls the C-compiled main routine, intercepts standard output streams in real-time, and reads back the repaired file.

Code Transformation: File Loading & Validation

In the native C environment, files are loaded via standard disk I/O. In our WebAssembly port, this is mapped to a virtual filesystem in the browser's sandbox:

C Source Code (SGFC main.c)
/* Load and validate file in C */
int main(int argc, char **argv) {
    SGFInfo *info;
    char *infile, *outfile;

    ParseArgs(argc, argv, &infile, &outfile);
    
    // Read SGF from disk path
    info = LoadSGF(infile);
    if (info) {
        CheckSGF(info);
        if (outfile)
            SaveSGF(info, outfile); // Save output
    }
    return 0;
}
JavaScript (Emscripten Web Port)
// Virtual FS & WASM wrapper in JS
function runWasmAuditBg() {
    // 1. Write file to virtual FS
    wasmModule.FS.writeFile('/input.sgf', sgfContent);

    try {
        // 2. Execute C main via callMain
        wasmModule.callMain([
            '-U', '-c', 
            '/input.sgf', '/output.sgf'
        ]);
    } catch (e) {}

    // 3. Read auto-fixed content back
    correctedSgfContent = wasmModule.FS.readFile(
        '/output.sgf', { encoding: 'utf8' }
    );
}

Code Transformation: Output & Stream Redirection

Instead of printing reports directly to the terminal stdout/stderr stream, the WebAssembly module redirects all output to custom JavaScript event handlers:

C Source Code (SGFC error.c)
/* Print error details in C */
void PrintError(int err_num, const char *msg) {
    // Prints directly to stdout/stderr stream
    fprintf(stderr, "Error %d: %s\n", err_num, msg);
}
JavaScript (Emscripten Web Port)
// Intercept stdout and stderr streams
createSGFC({
    // Intercept standard output stream
    print: function(text) { 
        wasmOutput.push(text); 
    },
    // Intercept standard error stream
    printErr: function(text) { 
        wasmOutput.push(text); 
    }
}).then(Module => {
    wasmModule = Module;
});

5. Move Phase Detection

We developed a dynamic phase classifier in JavaScript. By calculating move indexes and the density of stone placement on the board, it dynamically segments the game record into:

  1. Fuseki (Opening): Early game setup.
  2. Chuban (Middlegame): Tactical clashes and fight structures.
  3. Yose (Endgame): Territory definition.

6. Roadmap

Our roadmap outlines our progressive path to creating a comprehensive, standard-compliant SGF pipeline:

Phase 1: SGF Compliance Auditor (COMPLETED)

Allows users to drag/upload SGF files. Runs them through the official SGFC reference validator compiled to WebAssembly, displays statistical cards immediately, runs a real-time progress diagnostics playback, and displays structured tables of errors and warnings.

Phase 1.1: Compliance Editor (UPCOMING)

When a file is flagged as non-compliant, the application will redirect users to an interactive editing workspace. This space will list errors and provide automated fixes (e.g. deleting empty nodes, removing obsolete tags, correcting invalid formats) to export a 100% compliant SGF.

Phase 2: Term-to-Node Move Assigner (NEW / IN PROGRESS)

We are designing a system to assign metadata and tactical terms directly to individual move nodes (e.g., classifying a node as a "Tesuji", "Ko", "Sente", or "Joseki"). This assigns structured intelligence to the game record.

Phase 3: Phase Detector (COMPLETED)

Classifies the moves of a game into Fuseki, Chuban, and Yose. Built into the auditor pipeline and visualized dynamically.

Phase 4, 5, 6: Advanced Compliant Appending (UPCOMING)

We plan to generate advanced SGF files by appending modern analytical data (such as AI evaluation scores, win rates, and tactical coordinates) to standard records.

Core Constraint: The final advanced SGF file must remain 100% compliant with SGF FF[4] Go specifications. We will achieve this by encapsulating our custom data inside valid, standard-compliant channels (e.g., standard comment structures or private properties prefix-guarded by X-), allowing modern readers to decode the extra metrics while ensuring legacy SGF readers do not crash.
