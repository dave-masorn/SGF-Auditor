A few things looked odd, and of course the new properties such as arrows were not visible. Here are my notes: newlines: the new '\n' newlines appear as little boxes (the symbol for unprintable characters) on the Mac. '\r' works as a newline. renumbering, move number in variations: some moves get labeled '-2' and '-1'. That cannot be the intention. What's happening here? long labels: Based on my experience, I think that even 4 is too long, and 3 can be difficult to display. Maybe you have a bigger screen than me? :) One radical new thought: (probably too radical) Since annotations goodmove, badmove, interesting, doubtful, and even are mutually exclusive, they could be merged into one single property with values from say 0 (worst) to 9 (best). These would be mapped to match the old properties. 0 = very bad, 1 = bad, 2 = very doubtful, 3 = doubtful, etc. File SGF file format FF[4]: Game info in merged trees: This is different from the old version. You allow splitting of the game info of a game into single info properties along the path to the root. In SGB there may only be one node for each game that contains all game info properties for this path. While your new rule has a certain logic to it, keeping all info for a game in one node seems simpler to use for me. Just imagine adding and deleting games in a collection. Your way, you have to move info properties around in the tree. It might get quite complex if you have 732 games where Takemiya is black, then you add a few games by other players. The old way this needs no special treatment. You write: Game-info properties may only appear once in any path within the tree, i.e. if a game-info propererty gets set in one node it must not appear ^^^^^ in any subtrees of this node. I suggest changing it to: A node containing Game-info properties is called a game-info node. There may only be one game-info node on any path within the tree, i.e. if some game-info properties occur in one node there may not be any further game-info properties in following nodes: a) on the path from the root of the merged game to this node. b) in the subtree below this node. Instead of 'there may be only one' there could be the stronger 'there must be exactly one'. 'Triple' type. This is a slight misnomer. After all, a triple is something like (a,b,c). I have heard the term 'tristate' used, but don't know whether that is good English. Compressed point lists The propvalue type: "list of point" or "elist of point" may be compressed. Compressing is done by specifying rectangles instead of listing every single point in the rectangle. Rectangles are specified by using the upper left and lower left corner of the rectangle. ^^^^^^^^^^^ This should be 'lower right' File SGF Go properties: Rank: I don't like giving professional ranks as 'p'. This is a IGS only convention, not used anywhere else AFAIK. 'Dan' is much more usual. Property: TC Function: Gives the difference of black minus white territory.
